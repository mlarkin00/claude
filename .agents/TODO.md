## P1 — Important / Unblocking

- [ ] **[P1]** Neither sidecar is registered or run by Antigravity — both are dead files as shipped. `agy` loads sidecars from **`<config>/sidecars/<id>/sidecar.json`** (i.e. `~/.gemini/config/sidecars/`), one directory per sidecar, per the spec embedded in the binary; it does not look inside `plugins/<name>/sidecars/`, which is where `agy plugin install` puts ours. Confirmed against a live install: `~/.gemini/config/sidecars/` does not exist, no `sidecar_data/` runtime directory was ever created, no sidecar process is running, and the installer never lists sidecars as a component at all. The visible consequence: `skill-usage` counted a real skill invocation in a live session (`systematic-debugging` -> 1) but the counts file stays **uncommitted**, because committing and pushing is `sidecars/sync-usage`'s job. `active-skills`' `check-updates` never runs either — academic while it polls a 404 (below), but broken independently. **Fix:** install sidecars into `~/.gemini/config/sidecars/<id>/` (symlink or copy from the plugin — the execution CWD is the sidecar's own folder, which `check_updates.py` already assumes), or drop them and move the work into a hook. A plugin cannot ship a working sidecar by placing it under `sidecars/` and expecting `agy plugin install` to find it.

- [ ] **[P1]** Plugin agents install but cannot be invoked under Antigravity — all eight (`agent-memory` 3, `memory-bank` 1, `llm-wiki` 4) are copied to `plugins/<name>/agents/` and counted at install (`agents : 3 processed`), but `agy agents` lists nothing and a live session asked directly answered **"NO PLUGIN SUBAGENTS"**, offering only the built-in `research` and `self`. They are not converted to skills either. The agent layer of three plugins is therefore inert on that runtime: anything that says "run the bootstrap-memory agent" — including `verify-memory.sh`'s own remediation message — has no way to happen there. **Fix:** decide whether these become skills for Antigravity (the only component type that demonstrably loads) or whether the plugins document agents as Claude-only. Pair with the `agent-memory` agent-frontmatter item below, since fixing that is wasted effort if the agents are unreachable anyway.

- [ ] **[P1]** `agent-memory`'s three agents load with **empty metadata** — `agents/bootstrap-memory.md`, `memory-puller.md`, `memory-pusher.md` all fail YAML frontmatter parsing: the `<example>` blocks sit *inside* the `---` fences, after the `description:` string has closed, so the parser hits raw XML at the top level of the mapping (`while scanning a simple key`). Flagged by `claude plugin validate agent-memory` and confirmed independently with PyYAML. **Confirmed live**, not theoretical — the session agent roster shows `agent-memory:bootstrap-memory: Agent from agent-memory plugin (Tools: All tools)` (placeholder description, no tool restriction) against a correctly-parsed sibling, `memory-bank:bootstrap-memory-bank: Use when setting up… (Tools: Bash, Read, Edit)`. Dropped on all three: the `description` (so the model cannot tell when to invoke them), `tools: [Bash]` (**they run with every tool instead**), and `model: haiku` (they run on the session model). **Fix:** move the `<example>` blocks below the closing `---`, matching `memory-bank/agents/bootstrap-memory-bank.md` and `llm-wiki/agents/*.md`. Re-verify with `claude plugin validate agent-memory` **and** by confirming the roster shows the real description and `(Tools: Bash)`.

- [ ] **[P1]** `llm-wiki` registers **10 duplicate skill names** in Claude Code — `claude plugin details llm-wiki` reports `Skills (27)` with `enrich, index, ingest, init, lint, log, query, stats, validate, visualize` each listed twice, because the plugin ships both `commands/<n>.md` and `skills/<n>/SKILL.md` for all ten. Claude Code now surfaces commands as skills, so the two collide and **the command wins**: the session skill roster shows the *command's* description for `llm-wiki:stats` and `llm-wiki:enrich`, meaning the `SKILL.md` bodies — the actual instructions — never load. The command bodies then say things like "Load the `stats` skill for interpretation", a hop that cannot resolve because the name routes back to the command. Costs ~700 always-on tokens for the shadowed copies. Antigravity no longer has this problem, though not by design: since `llm-wiki` gained a root `plugin.json` with the P0 fix, `agy` takes the native path and does not convert the commands at all, so only the 17 real skills load there. **Fix:** pick one surface per name. Either delete the ten thin `commands/*.md` wrappers and let the skills carry `/llm-wiki:<n>` invocation, or keep the commands as the entry point and fold the skill bodies into them. Deleting the commands is the smaller change and preserves the richer content.

- [ ] **[P1]** `active-skills`' update-check sidecar polls a URL that is now a permanent 404 — `sidecars/check-updates/check_updates.py:62` fetches `https://raw.githubusercontent.com/mlarkin00/active-skills/main/plugin.json`, but the source repo went skills-only in the 2026-07-22 restructure and has no manifest (verified HTTP 404; the repo root holds 39 skill dirs plus `AGENTS.md` and `README.md`). It fails soft, so it will log `Update check failed. Preserving last known state` forever; its cached state already reads `local=0.1.4, remote=0.1.5` against an actual 0.2.7. **Fix:** point it at this repo's manifest (`raw.githubusercontent.com/mlarkin00/plugins/main/active-skills/plugin.json`), since the version now lives here — or delete the sidecar if nothing consumes its output. Note `tests/test_check_updates.py` passes today because it mocks the fetch, so the suite will not catch the next URL rot; assert the URL resolves, or at least that it targets this repo.
