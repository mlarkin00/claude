# Database Migration Skill

This skill provides a structured approach to managing database schema changes reliably and safely. It covers creating, applying, and verifying migrations across various tools.

## 🚨 MANDATORY RULES 🚨

1.  **NEVER** perform destructive schema changes (DROP, TRUNCATE, DELETE) without explicit user confirmation and a dry run.
2.  **ALWAYS** recommend or perform a dry run before applying migrations to production-like environments.
3.  **ALWAYS** verify the database state after a migration is applied.
4.  **NEVER** skip versioning. Every schema change MUST be part of a versioned migration script.
5.  **ALWAYS** check for existing migrations to avoid version conflicts or duplicate applications.

## Core Workflow

### 1. Identify the Tool and Environment
- [ ] Determine which migration tool is in use (e.g., Flyway, Liquibase, Alembic, Django, Rails).
- [ ] Identify the target database engine and environment (dev, staging, prod).
- [ ] Verify access and permissions to run migrations.

### 2. Creating a Migration
When creating a new migration script:
- [ ] Generate or write the migration script following naming and versioning conventions.
- [ ] Review the script for idempotency and efficiency.
- [ ] Ensure the script is descriptive (e.g., `V2__add_user_email_index.sql`).

### 3. Applying Migrations
- [ ] **MANDATORY**: Run a dry run or "info" command first to see what will be applied.
- [ ] Apply migrations using the tool's CLI or API.
- [ ] Capture and report any errors immediately.

### 4. Verification
After applying migrations:
- [ ] Check the tool's "history" or "status" table (e.g., `flyway_schema_history`, `alembic_version`).
- [ ] Verify the schema changes actually took effect by inspecting the database schema directly.

### 5. Rollback
If a migration fails or needs to be undone:
- [ ] Use the tool's built-in rollback/downgrade functionality if available.
- [ ] Verify the database state after rollback.

## Tool-Specific Guidance

### Flyway
- **Status**: `flyway info`
- **Migrate**: `flyway migrate`
- **Repair**: `flyway repair` (use if checksums mismatch)

### Alembic (Python)
- **New Migration**: `alembic revision -m "description"` or `alembic revision --autogenerate -m "description"`
- **Migrate**: `alembic upgrade head`
- **Rollback**: `alembic downgrade -1`

### Django
- **New Migration**: `python manage.py makemigrations`
- **Migrate**: `python manage.py migrate`
- **Status**: `python manage.py showmigrations`

## Gotchas & Anti-Patterns

| Excuse | Reality |
| :--- | :--- |
| "A manual schema change is faster for simple additions." | Manual changes lead to drift and versioning hell. Always use a migration script. |
| "Autogenerate caught everything." | Autogenerate is a starting point, not a final script. Always review autogenerated code. |
| "Editing the existing migration file is easier than creating a new one." | NEVER edit a migration that has already been applied. Create a new migration instead. |
| "Rollbacks are not necessary for this simple change." | Every migration MUST have a plan for what to do if it fails. |

## Success Criteria
- Migration script is created and follows naming conventions.
- Dry run is performed and reviewed.
- Migration is applied without errors.
- Database state is verified post-migration.
